<a id="问题一"></a>
## 问题一：小游戏 / 日历 / 属性 / 任务达成后，怎么跳到指定脚本或指定对话？

**全部可以，而且不用写任何代码，纯剧本就能实现。** 核心是理解两件事：

1. **本文件内跳转用 `jump 标签`，跨文件跳转用 `chapter 文件名`**。
2. **本项目的"万物皆 flag"设计**：属性（`stat`）、月份（`time`）、任务状态（`quest`）、
   道具（商店购买）……底层全部是 VNFlags 里的整型变量，所以 **`if` 条件对它们全部通用**。

### 1.1 两种跳转命令

```
# 同一个 .vn.txt 内：跳到 label
jump 告白场景

# 跳到另一个剧本文件（从该文件第一行开始执行）
chapter 第二章
```

`chapter` 的前提：目标 TextAsset 必须拖进场景里 `VNScriptRunner` 的
**Chapters** 列表登记（Inspector 上）。文件名不区分大小写、可省略 `.vn.txt` 后缀。
flag / 属性 / 任务状态是全局的，切文件**不会丢失**。

### 1.2 跳到"另一个文件的指定对话"——路由器套路

`chapter` 只能从目标文件的开头执行。想直达文件中间的某一场，用
**"入口 flag + 文件头路由"**套路：

跳转方（当前文件）：

```
flag 进入点 3
chapter 十二月篇
```

目标文件 `十二月篇.vn.txt` 的开头：

```
# ===== 文件头路由：按进入点分发 =====
if 进入点==1 jump 期末考试
if 进入点==2 jump 寒假开始
if 进入点==3 jump 圣诞约会
# 没设进入点 = 从头正常播

label 圣诞约会
bg 夜晚街道 transition:NoiseDissolve
...
```

> 建议每个章节文件的开头都保留这样一段路由区，其他文件就都能"点名"跳进来。
> 用完记得 `flag 进入点 0` 清零，避免下次误触发。

### 1.3 小游戏（event 事件）结束后跳转

`event` 的结果行本来就支持 `-> 标签`（写法同 choice 选项）：

```
event qte time:3 target:12 title:期末考试冲刺！
* success flag:考试结果+1 -> 考试通过
* fail -> 考试挂科

label 考试通过
: （太好了，及格了！）
chapter 寒假篇        # 结果行跳到的 label 里再 chapter，即可跨文件
```

### 1.4 日历到达某一天（某月）时跳转

`time` 命令把月份写进名为 `月份` 的 flag、剩余月数写进 `剩余月数`，
所以直接用 `if` 检查即可。标准写法是在**过月之后立刻检查**：

```
label 月末
time pass
if 月份==12 jump 圣诞活动     # 12 月一到就进活动
if 剩余月数<=0 jump 最终结局   # 养成期结束
jump 月初

label 圣诞活动
flag 进入点 1
chapter 节日活动集       # 也可以跳到专门的活动剧本文件
```

### 1.5 属性达标后开启 / 跳转

属性就是 flag，直接比较：

```
# 属性到 60 才开放的隐藏选项（不满足时选项隐藏）
choice
* 参加歌唱大赛 if:魅力>=60 -> 歌唱大赛
* 继续练习 -> 行动

# 或者剧情走到检查点时自动分歧
if 智力>=80 jump 保送线
if 金钱>=2000 jump 富豪线
```

### 1.6 完成任务后跳转

`quest` 把状态写进 flag `任务_<id>`（0=未接取 / 1..n=进行中阶段 /
100=完成 / -1=失败）：

```
if 任务_告白大作战==100 jump 后日谈       # 已完成
if 任务_告白大作战==-1 jump 补救剧情      # 已失败
if 任务_告白大作战>=1 jump 进行中提示     # 接了还没做完
```

### 1.7 一张速查表

| 想要的触发 | 条件写法（都用 `if 条件 jump 标签`） |
|---|---|
| 小游戏成功/失败 | event 结果行直接 `-> 标签`，或结果行写 flag 后续再 if |
| 到了某个月 | `if 月份==12 jump …` |
| 养成期结束 | `if 剩余月数<=0 jump …` |
| 属性达标 | `if 魅力>=60 jump …` |
| 有某道具 | `if 道具_蝴蝶结发饰>=1 jump …` |
| 任务完成 | `if 任务_某任务==100 jump …` |
| 去过某地点 | `if 去过_天台 jump …`（map 模块自动写） |
| 跨文件 + 指定场景 | `flag 进入点 N` + `chapter 文件` + 目标文件头路由 |

## 问题二：这个游戏的 Game Loop 应该是怎么样的？现在的系统支持吗？

### 2.1 两类游戏的典型 Loop

**传统 Galgame（纯 ADV）**——线性阅读为主：

```
共通线（1~N 章顺序阅读）
   → 关键选项累积好感度 flag
   → 分歧点：if 好感度 检查 → 进入某角色个人线（chapter 切文件）
   → 个人线内再按选择分 好结局 / 普通结局 / 坏结局
   → 通关解锁（CG 鉴赏、周目要素）
```

**模拟养成（《火山的女儿》《美少女梦工厂》型）**——循环驱动：

```
┌─→ 月初/周初（time pass：行动力回满、日历前进）
│      ↓
│   玩家安排行动（choice：打工/学习/训练/逛街…，消耗行动力）
│      ↓
│   执行行动（演出 + stat 增减 ± 小游戏 event ± 随机事件）
│      ↓
│   行动力用完 → 月末结算
│      ↓
│   检查点：固定剧情事件（某月触发）/ 属性触发事件 / 任务进度
└──────┘
        ↓（剩余月数归零）
     结局判定（按最终属性/flag 分发多结局）
```

商业养成游戏基本都是这个骨架，差别只在"行动"的花样（选课、社交、地图移动）
和结算的复杂度（随机结果、连锁事件、好感度事件插入）。

### 2.2 现在的系统直接支持吗？——支持，已有可跑的完整示例

`Assets/Scenarios/RaisingDemo.vn.txt` 就是一个最小完整养成 Loop，
用到的全部是现成命令：

| Loop 环节 | 用的机制 |
|---|---|
| 月初回满行动力、日历前进 | `time pass`（自动回满 `行动力`、`月份`+1、`剩余月数`-1） |
| 行动菜单 | `choice` + `cost:行动力-2`（付不起自动置灰）+ `if:` 条件显隐 |
| 行动结果 | `stat 金钱 +150` / `stat 压力 +10`（自动钳制+飘字） |
| 小游戏行动 | `event qte …` / `event shop id:服装店` / `event map …` |
| 循环 | `label 月初` … `jump 月初` |
| 固定/触发事件 | `if 月份==X jump …` / `if 属性>=N jump …`（见问题一） |
| 结局判定 | 剩余月数归零后按属性 if 链分发 |
| 存档 | flag 全量入档，Loop 任意台词处可存 |

**推荐的文件组织**（Loop 骨架与剧情内容分离）：

```
Assets/Scenarios/
├── 主循环.vn.txt        ← 月初/行动菜单/月末/结局判定（骨架，改动少）
├── 第一章.vn.txt         ← 开场剧情，结尾 chapter 主循环
├── 节日活动集.vn.txt     ← 固定日期事件（文件头路由，见问题一 1.2）
├── 好感事件_小雪.vn.txt  ← 属性/好感触发的角色事件
└── 结局集.vn.txt         ← 各结局演出
```

主循环里只写"检查 + chapter 派发"，剧情本体放各自文件，
写作和 Git 协作都更干净。

### 2.3 目前的两个缺口（提前知道）

1. **没有内建随机数命令**——现在的行动结果是固定数值。想要"打工有概率
   大成功/失败"，要么用 `event qte`（把随机换成玩家操作），要么加一个小扩展
   （如 `flag 运气 rand:1-100` 或日程模块内部掷骰，见问题四）。
2. **周计划排程 UI 不存在**——现在是"每次行动选一次 choice"。一次排好
   一周再连播需要新写一个事件模块（可行性与设计见问题四）。

<a id="问题四"></a>
## 问题四：能做"排好一周日程 → 逐一执行 → 随机产出失败/普通/成功/大成功"吗？

（参考截图：《火山的女儿》式选课界面、训练执行演出、COMPLETE 结算加属性）

**结论：可以实现，架构完全兼容，但需要开发一个新的事件模块 + 一个随机数小扩展。**
这正是 `VNEventModule` 玩法事件接口设计时预留的扩展场景（和 qte/map/shop 同类）。

### 4.1 现在立刻能做到的近似版（零开发）

RaisingDemo 的模式已经是"单步版"：每次行动选一个 → 演出 → 加属性 → 循环。
和截图的差别只是"一次排 N 个再连播"和"结果随机分级"。

### 4.2 完整版的建议设计（新模块 `event plan`）

**① 排程模块**（新写 `VNPlanModule`，注册 id 为 `plan`）：

```
event plan slots:7 pool:打工,学习,剑术训练,休息 title:安排这一周
```

- 弹出日程面板：左侧候选行动（可显示预期收益/消耗），右侧 7 个格子；
- 玩家把行动填进格子，确认后模块把结果写进 flag：`日程_1=行动编号` … `日程_7=…`；
- 遵守模块三铁律（不碰舞台 / unscaled 计时 / SetLink），UI 风格参照
  VNShopModule 的面板做法。

**② 随机数扩展**（二选一，建议都做，工作量都很小）：

- `flag 运气 rand:1-100`——给 flag 命令加 rand 参数（VNScriptRunner 的
  flag case 里十几行），之后剧本可写：

```
label 执行_打工
flag 运气 rand:1-100
if 运气<=10 jump 打工_失败       # 10% 失败
if 运气<=80 jump 打工_普通       # 70% 普通
if 运气<=95 jump 打工_成功       # 15% 成功
jump 打工_大成功                  # 5% 大成功
```

- 或者由执行模块内部掷骰（结果行 `* great / * good / * normal / * fail`），
  概率可受属性影响（体力高→失败率降），更接近《火山的女儿》。

**③ 执行循环**（纯剧本，逐格播放）：

```
label 执行日程
flag 当前格 +1
if 当前格>7 jump 周末结算
if 日程_1==1 jump 执行_打工     # 按当前格读对应 flag 派发
...
```

**④ 结果演出**：GOOD! / COMPLETE! 弹窗用现成组件拼：`transition WhiteFlash` +
`se 成功音效` + Q 版立绘 `show` + `stat` 飘字（VNStatsHud 自带 +N 飘字，
截图左侧那种增益列表效果已经有了）。想要专门的结算大弹窗，可以做成
执行模块 UI 的一部分。

### 4.3 工作量评估

| 项 | 量级 |
|---|---|
| `flag … rand:` 随机数 | 很小（一次会话内完成） |
| 排程面板模块 VNPlanModule | 中等（一至两次会话，UI 是大头） |
| 执行/结算演出模块（分级结果+概率受属性影响） | 中等 |
| 纯剧本连播循环 | 零代码，写剧本即可 |

建议顺序：先加 `rand:`（立刻能玩随机结果），再做排程面板，最后做结算演出美化。

<a id="问题五"></a>
## 问题五：程序生成的 UI 之后想换美术风格，可能吗？最佳做法是什么？

**可能，而且当初就是按"占位、可替换"设计的。** 对话框、Config、存读档、
选项面板等都是标准 uGUI（Image + TMP），程序生成只是"谁来摆"的问题，
不影响"摆什么图"。三条路线，按投入从小到大：

### 路线 A：主题资产（VNTheme ScriptableObject）——推荐首选

新建一个 `VNTheme` 资产，集中存：九宫格面板底图、按钮图、名牌图、箭头图、
配色表、字体引用。各 UI 组件生成时先查主题资产，**有图用图、没图回退到
现在的程序化贴图**。

- 优点：与现有"生成器一键重建场景"工作流**完全兼容**（重建后皮肤不丢）；
  换风格 = 换一个资产文件；可以做多套主题切换。
- 缺点：只换"皮"（图、色、字体），不改布局结构。
- 适合：找美术出一套 UI 素材图后整体换皮——**90% 的换风格需求到此为止**。

### 路线 B：生成后转 Prefab

生成器跑一次 → 把 `DialogueBox`、`SaveLoadPanel` 等从场景拖成 Prefab →
生成器改为"有 Prefab 就实例化 Prefab，没有才程序生成"。

- 优点：可以在编辑器里**自由改布局**（挪位置、加装饰、换层级），美术可直接参与。
- 缺点：Prefab 与代码字段要保持连线（组件引用的子物体名字/结构不能乱改）；
  之后代码给 UI 加新元素时，Prefab 需要手动同步更新，**重建场景不再能自动
  修复这些 UI**。
- 适合：正式美术风格定稿后，对话框这类"门面"UI 的深度定制。

### 路线 C：完全手工 UI

抛弃生成，全部手搭。不推荐——失去一键重建能力，且本项目 UI 数量已不少。

### 建议的实施节奏

1. **现阶段**（素材未定）：什么都不用做，程序化 UI 继续当占位。
2. **拿到 UI 素材包时**：先做路线 A 的 VNTheme（一次会话可完成骨架），
   把面板底图/按钮图/配色换掉——视觉上就已经"换了游戏"。
3. **风格定稿要精修时**：对话框、标题界面等门面 UI 走路线 B 转 Prefab，
   功能性面板（Config/存读档/物品栏）留在路线 A 就够。

<a id="问题六"></a>
## 问题六：除了 UI，还有什么是程序生成的？这样做好吗？

### 6.1 程序生成清单

| 类别 | 内容 | 换素材的难度 |
|---|---|---|
| UI 面板 | 对话框/选项/Config/存读档/工具条/Toast/各 HUD/事件模块面板 | 见问题五 |
| 程序化贴图 | VNProceduralTextures：柔圆粒子、四芒星、光晕、光束、花瓣、圆环、圆角面板等所有特效贴图 | 容易——粒子和特效组件都持有 Sprite/Texture 字段，可直接替换成美术图 |
| 材质资产 | `Assets/VNEffects/Materials/`（生成器创建，shader 特效用） | 一般不用换，换 shader 参数即可 |
| 粒子系统 | 8 种环境粒子全部代码配置（尘埃/雨/雪/萤火虫…） | 改参数要改代码；但贴图可换 |
| 演示场景本身 | VNEffectsDemo / VNScriptDemo 由生成器一键重建 | 这是优点：场景坏了随时重来 |
| 字体资产 | 动态 SDF（但已是持久化资产，不算运行时生成） | 换字体 = 换 VNFont 引用 |

**不是**程序生成、本来就靠资产驱动的：立绘/背景/CG（图片文件）、音频（AudioClip）、
角色定义（VNCharacterDef）、属性/商店/任务定义（VNStatDef/VNShopDef/VNQuestDef）、
剧本（.vn.txt）、翻译表。**内容层全部是数据文件，这部分随时可换可加。**

### 6.2 这样做好吗？

**当前阶段是正确选择**：

- 零美术依赖，一个人 + AI 就能把全部系统跑通；
- 场景可一键重建 = 没有"场景文件坏了/连线丢了"这种不可恢复灾难；
- 纯代码生成对 Git 和 AI 协作友好（场景二进制 diff 最小化）。

**代价与对策**：

- 改视觉要改代码 → 对策就是问题五的 VNTheme/Prefab 化，**按需逐步**把
  "视觉常量"从代码挪到资产，不必一次性重构；
- 手动改了生成场景会被下次重建覆盖 → 已定稿的东西转 Prefab 固化；
- 粒子/特效参数在代码里 → 这些本来就是"演出逻辑"，放代码反而好维护
  （有 Git 历史、可搜索、可注释）。

一句话：**逻辑和演出留在代码，皮肤和内容逐步资产化**——现在的方向不用推翻。

<a id="问题七"></a>
## 问题七：追加小功能容易吗？

**容易，这套架构对小功能扩展非常友好。** 原因：命令是一张 switch 表
（VNScriptRunner），全局状态只有一种（VNFlags，还带 Changed 事件），
UI 都是独立小组件。你举的几个例子逐个看：

### ① 日期变更时插一条"日期横幅从左滑入"演出

- 系统已有 `VNFlags.Changed` 事件，VNCalendarHud 就是靠它刷新的。
- 做法：给 VNCalendarHud（或新建 VNDateBanner）加一段"月份变化时播放
  滑入横幅 + SE"的 DOTween 动画，`time pass` 时自动触发，不用改剧本语法。
- 量级：**小**（一次会话）。

### ② 转场画面 fade in/out + 音效

- **现在就能做，零开发**，纯剧本：

```
se 翻页声 @
transition WhiteFlash        # 或 NoiseDissolve / Eyelid 等 8 种
```

- 想要"纯黑淡入淡出"这种新转场类型：VNScreenTransition 加一个枚举 + 一段
  实现，量级**小**。

### ③ 新属性：社交度、享受度、别的角色的好感度、友情度……

- **零代码，现在就支持。** 属性本质是 flag：
  - 直接用：`stat 社交度 +5`、`if 好感度_小雪>=30 jump …`——没有定义资产
    也照常运作（只是不钳制、不上 HUD）；
  - 想要上顶栏 HUD / 钳制范围 / 等级显示：在 `Assets/VNEffects/Stats/` 右键
    **Create → VN → Stat Definition** 建一个资产、拖进 VNStatsHud 列表即可。
- 每个角色的好感度就用命名约定 `好感度_小雪`、`友情_亚里沙`，if/存档/结局
  判定全部自动可用。

### ④ 完成任务后弹结果弹窗 + 一段演出

- 演出部分现在就能写在剧本里（`quest done` 后接 transition/sakura/台词）；
- "自动弹结算窗"：给 VNQuestLog 加一个完成时的弹窗（它已经监听任务状态，
  VNToast 弹提示的位置换成大弹窗即可），量级**小～中**。

### 追加一条新剧本命令的通用流程（给你个直觉）

1. 功能组件写好（挂 VNStage 或独立管理器）；
2. VNScriptRunner 的 switch 加一个 `case "命令名"`（解析是通用的，多数不用动 Parser）;
3. 需要入存档就在快照/恢复里加一笔；需要编辑器支持就在 VNScenarioSchema 登记；
4. 重建演示场景 + 在 Demo 剧本加示例行。

历史数据参考：stat 系统、time 日程、商店模块各自都是一次会话内完成合并的。
**结论：小功能"一个想法一个分支一次会话"是这个项目的常态节奏，放心提需求。**
