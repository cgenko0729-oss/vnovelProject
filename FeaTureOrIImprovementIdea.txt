  # 二、Flag 系统

  当前实现位于 /D:/UnityAiProject/My project/Assets/Scripts/VNEffects/Script/VNFlags.cs:15。

  它本质是一个全局的：

  字符串名称 → 整数数值

  例如：

  - 已看_圣诞 = 1
  - 好感度_小雪 = 35
  - 任务_告白大作战 = 100
  - 道具_药水 = 3
  - 入口_节日活动 = 2

  ## 优点

  ### 1. 极其通用

  一个整数可以扮演很多角色：

  - 布尔值：0/1
  - 次数：访问次数、失败次数
  - 阶段：主线进度、任务阶段
  - 枚举：0=无、1=成功、2=失败
  - 资源：金钱、道具数量
  - 路由参数：进入某文件的哪个入口
  - 一次性标记：是否领取奖励
  - 周目记录：是否通关某路线

  这种设计让你不用为每一种用途都建立新系统。

  ### 2. 系统之间天然互通

  地图模块可以写入 去过_天台，任务系统不需要认识地图模块，只要剧本检查：

  if 去过_天台>=3 jump 推进任务

  商店写入 道具_蝴蝶结发饰，好感事件就可以直接检查它。

  这是当前架构最有价值的部分。

  ### 3. 存档成本很低

  整个字典直接进入存档，所以新增普通 Flag 时，一般不需要修改存档结构。

  ### 4. 非常适合文本剧本和人工调试

  Flag 名可以使用中文，编剧看到 已看_圣诞 会比看到某个数字编号更容易理解。

  ### 5. 性能完全不是问题

  即使以后有几千个 Flag，对字典查询来说也不是什么性能压力。真正的问题会是管理和正确性。

  ## 缺点

  ### 1. 只有整数

  目前不能自然表示：

  - 字符串
  - 浮点数
  - 日期对象
  - 列表
  - 角色集合
  - 日程数组
  - 复杂任务状态
  - 临时结构化结果

  例如“当前服装名称”“本周7个日程”“最近三次行动结果”都不适合硬塞进单个整数。

  ### 2. 不存在与数值0无法区分

  读取一个从未创建的 Flag，也会得到0。

  因此系统无法区分：

  - 这个变量不存在
  - 这个变量明确被设置为0
  - 旧存档还没有这个新变量
  - 新游戏初始化故意把它设成0

  目前 VNStatDef 的初始化机制会检查字典里是否存在键，但普通剧本条件无法区分这两种状态。

  ### 3. 全局命名空间容易碰撞

  这些名字都存在同一个字典里：

  - 进入点
  - 结果
  - 阶段
  - 成功
  - 次数

  如果多个系统使用过于笼统的名称，很容易互相覆盖。

  你现在采用：

  - 入口_节日活动
  - 入口_小雪事件
  - 已看_第2章
  - 任务_告白大作战

  这种带前缀的方式更安全。

  ### 4. 全靠字符串，拼错不会编译失败

  例如：

  好感度_小雪
  好感度_小雷
  好感度小雪

  系统会认为这是三个完全不同的变量。

  这是内容量扩大后最常见、也最难排查的问题之一。

  ### 5. 重命名会破坏旧存档

  假设正式版把：

  好感度_小雪

  改成：

  爱情度_小雪

  旧存档里仍然只有旧名字，新系统就会把新名字当成0。

  ### 6. 任何系统都能任意修改任何 Flag

  这虽然灵活，但没有权限边界。

  例如：

  - 剧本用 flag 直接修改属性，可以绕过 stat 的钳制和表现。
  - 剧本直接把任务状态写成37，任务系统无法判断这是什么意思。
  - 道具数量可能被写成负数。
  - 入口变量可能忘记清零。

  ### 7. Changed 事件信息不足

  当前 VNFlags.Changed 只告诉监听者“有变量变化了”，没有告诉它：

  - 哪个变量变了
  - 原值是多少
  - 新值是多少
  - 谁修改的
  - 修改原因是什么

  所以属性 HUD、日历、任务等收到通知后，只能自行重新读取。

  ### 8. 没有事务

  一次行动可能同时修改：

  - 金钱 -100
  - 好感 +5
  - 压力 -3
  - 道具 +1

  当前这些是四次独立变化。如果中间失败，可能留下半套结果；UI 也可能连续弹出多个提示。

  # 三、Label、Jump、Chapter 系统

  相关逻辑主要在 /D:/UnityAiProject/My project/Assets/Scripts/VNEffects/Script/VNScriptRunner.cs:443。

  ## 优点

  ### 1. 简单直接

  label 是地点，jump 是跳转。编剧很容易理解。

  ### 2. 支持向前和向后跳转

  运行前会预扫描当前文件全部标签，因此可以跳到后面尚未执行的标签。

  ### 3. 比行号稳定

  剧本插入或删除几十行后，标签仍然有效。

  ### 4. 非常适合循环结构

  可以实现：

  - 行动菜单循环
  - 月循环
  - 商店返回
  - 失败重试
  - 多结局派发
  - 文件头路由

  ## 缺点

  ### 1. Jump 只能跳当前文件

  跨文件必须先 chapter，而 chapter 只能从目标文件开头进入。

  因此才需要：

  入口 Flag → chapter → 文件头 if 路由

  这个套路有效，但内容多了会显得繁琐。

  ### 2. 没有调用和返回

  现在只有“跳过去”，没有：

  - 调用一个公共片段
  - 执行完自动返回原处
  - 给公共片段传参数

  因此重复的小剧情可能需要复制，或使用大量入口 Flag 模拟。

  ### 3. 没有局部标签或命名空间

  同一文件不能定义两个同名标签。

  大型文件容易出现：

  - 结束
  - 结束2
  - 真正结束
  - 结束返回
  - 结束返回2

  ### 4. 容易形成意大利面流程

  大量 jump 会让剧情变成网状结构，阅读文件时很难知道玩家从哪里来、接下来去哪。

  ### 5. 没有循环保护

  错误写法可能造成无台词、无等待的无限跳转，直接卡死一帧。

  ### 6. 跨文件引用缺少完整验证

  当前运行时能报告找不到章节或标签，但缺少全项目的：

  - 跨文件流程图
  - 不可达标签检查
  - 无出口路径检查
  - 死循环检查
  - 入口变量未清零检查

  ### 7. 中间行调试无法精确还原分支历史

  当前“从选中行播放”遇到 choice/jump/if/event 时，只能按文本顺序近似重建状态，不知道玩家之前实际走了哪条路。

  ## 潜力与改进方向

  1. 支持限定跳转：

  jump 节日活动集::圣诞约会

  这样可以直接跨文件跳标签，不再依赖入口 Flag。

  2. 保留入口 Flag 机制作为“带参数路由”。

  直接跨文件跳转适合固定目标；入口 Flag 适合一个文件有多个入口和附加状态。

  3. 增加 call/return：

  call 公共事件::获得奖励

  执行后自动回到原位置。

  4. 支持参数：

  call 通用送礼 角色:小雪 道具:蝴蝶结 好感:10

  5. 增加标签属性：

  - 一次性
  - 自动存档点
  - 检查点
  - 调试入口
  - 优先级
  - 章节标题
  - 是否允许回滚

  6. 增加稳定节点编号。

  存档不要只依赖命令下标，而应保存稳定的剧情节点编号。

  7. 增加流程图编辑器：

  - 显示文件、标签、选择和章节关系
  - 点击节点跳到剧本
  - 标记不可达节点
  - 标记无出口节点
  - 显示读取/写入的 Flag

  8. 增加无限循环保护：

  同一帧连续执行超过一定数量命令时暂停并报告跳转链。

  # 四、Event 事件系统

  相关基础位于 /D:/UnityAiProject/My project/Assets/Scripts/VNEffects/Script/VNEventModule.cs、/D:/
  UnityAiProject/My project/Assets/Scripts/VNEffects/Script/VNEventRegistry.cs。

  ## 优点

  ### 1. 剧本和玩法解耦得很好

  剧本只知道：

  - 模块ID

  当前同一个接口已经承载：

  - QTE
  - 地图
  - 商店

  说明接口不是只为某个单一小游戏写死的。

  ### 3. 结果能够直接驱动剧情

  事件结果可以：

  - 写入 Flag
  - 跳转标签
  - 更新任务
  - 增加属性
  - 进入其他章节

  ### 4. 生命周期隔离较清楚

  事件期间会停止自动模式和快进，隐藏对话框，阻止剧本快捷键干扰玩法。

  ### 5. 很适合继续增加周计划系统

  周计划、考试、烹饪、约会、打工、战斗都可以作为事件模块接入。

  ## 缺点

  ### 1. 参数全部是字符串

  例如：

  time:3
  target:12

  模块自己负责解释。

  拼成 targte:12 时，解析器无法知道这是错误参数。

  ### 2. 结果名必须精确匹配

  模块返回 success，剧本写成 Success 或 成功，就无法匹配。

  ### 3. 缺少模块声明

  编辑器不知道一个模块：

  - 接受哪些参数
  - 参数类型是什么
  - 默认值是什么
  - 会返回哪些结果
  - 是否允许存档
  - 是否需要3D场景
  - 是否需要特定资源

  ### 4. 事件期间不能存档

  短QTE没有问题，但如果以后出现：

  - 15分钟战斗
  - 长时间经营
  - 一周日程连续执行
  - 卡牌关卡

  不能中途保存会成为明显限制。

  ### 5. 模块失败时可能顺序继续

  找不到模块、没有结果、结果拼错时，目前部分情况会警告后继续剧本。

  这可能隐藏严重配置错误。

  ### 6. 没有超时和异常恢复

  如果模块永远不调用完成回调，剧本会一直等待。

  ### 7. 一次只能运行一个事件

  目前这其实是合理限制，但未来必须明确：

  - 能不能嵌套事件
  - 事件中能不能打开任务日志
  - 事件中能不能再调用商店
  - 事件能不能暂时进入另一个场景

  ## 潜力与改进方向

  1. 给每个事件模块建立定义：

  - ID
  - 中文名称
  - 参数列表
  - 参数类型
  - 必填/可选
  - 默认值
  - 允许结果
  - 保存策略
  - 场景模式

  2. 让剧本编辑器根据定义自动生成控件。
  3. 静态验证结果名：

  模块声明只会返回 success/fail，剧本写了 cancel 就直接提示。

  4. 结果从字符串升级为结构：

  结果类型：success
  分数：85
  评级：great
  获得金钱：200
  技能经验：15
  特殊标记：首次通关

  然后剧本可以根据不同字段判断。

  5. 增加默认结果：

  当模块返回未接住的结果时，进入专门的默认分支，而不是静默继续。

  6. 增加取消和超时：

  - 玩家取消
  - 模块异常
  - 超过最大时间
  - 场景加载失败
  - 读档中断

  7. 增加可保存事件接口。

  短事件仍不保存；长事件可以提供自己的状态快照。

  8. 支持附加场景：

  - 淡出视觉小说画面
  - 加载3D小游戏场景
  - 执行玩法
  - 返回结构化结果
  - 卸载场景
  - 恢复舞台

  9. 加入确定性随机数。

  事件使用存档中的随机种子，防止无意的结果变化和读档刷结果。

  10. 增加事件历史：

  记录什么时候进入、参数、结果、耗时，方便调试任务和结局。

  - 格式化
  - 变化动画

  ### 3. 定义资产需要登记

  属性定义存在于资产，但 HUD 要知道它们。随着角色和属性增加，手动登记会变麻烦。

  ### 4. 没有临时修正

  现在不容易表达：

  - 装备让魅力+10
  - 生病让体力上限-20
  - 节日活动成功率+15%
  - 下周学习收益翻倍
  - 三回合内压力增长减少

  ### 5. 没有派生属性

  例如：

  考试能力 = 智力×0.7 + 感性×0.2 - 压力×0.3

  现在要手动在每个事件里重新计算。

  ### 6. 没有阈值跨越事件

  系统知道属性变化，但不会自动产生：

  - 好感第一次达到20
  - 压力第一次超过80
  - 智力从B升到A
  - 金钱首次达到1000

  目前需要在剧本检查点手动判断。

  ### 7. 人际关系会让普通属性面板膨胀

  如果以后有20个角色，每人有：

  - 好感度
  - 友情度
  - 信赖度
  - 嫉妒度

  全部塞进普通属性HUD会失控。

  ## 潜力与改进方向

  1. 建立属性注册库，自动加载所有 VNStatDef。
  2. 增加属性类别：

  - 基础能力
  - 资源
  - 心理状态
  - 技能
  - 角色关系
  - 隐藏数值

  3. 增加临时修正：

  基础魅力 40
  装备修正 +10
  节日加成 +5
  最终魅力 55

  4. 增加派生属性和公式。
  5. 增加阈值事件：

  属性跨越指定门槛时发送一次明确事件。

  6. 增加变化批次：

  一次行动的所有变化合并成一个结果窗口：

  - 金钱 +150
  - 压力 +10
  - 好感 +5
  - 行动力 -2

  7. 增加变化预览。

  玩家选择行动前看到预计收益、范围和风险。

  8. 增加属性历史图表。

  显示几个月内体力、智力、压力的变化趋势。

  9. 建立独立关系系统。

  底层仍可与 Flag 兼容，但使用专门的：

  - 角色关系定义
  - 关系等级
  - 关系事件
  - 关系界面
  - 好感变化提示
  - 角色路线锁定

  10. 增加衰减和恢复：

  - 每月压力自然下降
  - 长期不联系好感下降
  - 体力每周恢复
  - 熟练度长期不使用缓慢下降

  # 六、Choice 选项系统

  ## 优点

  ### 1. 已经超越普通分支按钮

  每个选项现在可以：

  - 条件显隐
  - 判断花费
  - 自动扣费
  - 写Flag
  - 跳转标签
  - 本地化显示

  ### 2. 条件隐藏和花费置灰语义清楚

  - 资格不足：隐藏
  - 钱不够：看得到但不能点

  这种区分是合理的。

  ### 3. 本地化按索引匹配

  逻辑不依赖翻译后的选项字符串，相对安全。

  ### 4. 可以自然连接养成系统

  很适合：

  - 属性资格
  - 道具资格
  - 好感资格
  - 行动力消耗
  - 金钱支付
  - 路线选择

  ## 缺点与重要风险

  ### 1. 只有一个条件

  无法自然表示：

  魅力>=50 且 好感度_小雪>=30

  只能提前用Flag或额外标签计算。

  ### 2. 只有一个花费

  不能一次要求：

  - 金钱100
  - 行动力2
  - 道具1个

  ### 3. 只有一个附带 Flag 操作

  复杂选择仍需要跳转到标签后执行多条命令。

  ### 4. 所有选项都隐藏时会强制全部显示

  这是当前一个值得优先修正的风险。

  为了避免卡死，运行时会把全部隐藏选项重新显示。这可能：

  - 提前暴露隐藏剧情
  - 让未解锁路线被玩家看见
  - 泄露秘密选项
  - 破坏剧情资格设计

  ### 5. 所有选项都付不起时会强制解禁

  这可能让玩家免费选择本来需要付费的内容，形成经济漏洞。

  更合理的方式是要求编剧提供一个免费返回选项，并在编辑器阶段报错。

  ### 6. 没有“显示但因资格不足禁用”

  有时希望玩家看到：

  参加舞会——需要魅力50

  而不是完全隐藏。

  ### 7. 没有一次性选项和已选记录

  需要手动写 已选_某选项。

  ### 8. 没有限时选择

  无法实现：

  - 5秒内回答
  - 超时默认沉默
  - 紧张场景快速判断

  ### 9. 没有多选和排序

  不适合：

  - 一次选择三门课程
  - 安排一周日程
  - 选择多个装备
  - 给多个角色分配行动

  ## 潜力与改进方向

  1. 分离三种条件：

  - showIf：是否显示
  - enableIf：是否可点击
  - reason：不可点击原因

  2. 支持多个条件和逻辑表达式。
  3. 支持多个成本，作为原子事务一次扣除。
  4. 支持多个效果：

  - 修改多个属性
  - 发放道具
  - 推进任务
  - 写多个Flag
  - 播放音效
  - 跳转或调用剧情

  5. 增加稳定选项ID。

  用于：

  - 已选记录
  - 统计
  - 本地化
  - 安全重命名
  - 周目继承

  6. 增加一次性、可重复、冷却选项。
  7. 增加限时选项和默认结果。
  8. 增加技能检定：

  成功率受魅力、压力、道具影响

  玩家可以看到概率，也可以隐藏概率。

  9. 增加结果预览：

  - 预计属性变化
  - 消耗
  - 风险
  - 可能奖励

  10. 增加选项顺序随机化。
  11. 增加多选模式。

  这会直接成为周计划、选课和队伍编成的基础。

  12. 改进防卡死策略：

  - 编辑器保存时要求至少一个无条件免费选项
  - 运行时若无可用选项，显示错误提示并进入显式备用标签
  - 不要自动公开或免费解锁全部选项

  ———

  # 七、Quest 任务系统

  当前任务状态写入 任务_<id>，主要实现见 /D:/UnityAiProject/My project/Assets/Scripts/VNEffects/Script/
  VNQuestLog.cs:65。

  ## 优点

  ### 1. 极其轻量

  不需要复杂任务数据库就能运行。

  ### 2. 与 Flag、存档和 If 自动兼容

  任务完成后可以直接：

  if 任务_告白大作战==100 jump 后日谈

  ### 3. 支持阶段、完成和失败

  已经能覆盖简单线性任务。

  ### 4. VNQuestDef 将文案和逻辑分开

  任务资产负责：

  - 标题
  - 描述
  - 阶段文本
  - 本地化显示

  状态仍然在存档里。

  ### 5. 没有定义资产也能工作

  这对原型和快速写剧情很方便。

  ## 缺点

  ### 1. 使用魔法数字表达状态

  - -1
  - 0
  - 1～n
  - 100

  简单，但不够自解释。

  ### 2. 只能表达单线阶段

  不适合：

  - 多个目标同时进行
  - 可选目标
  - 分支任务
  - 失败后补救
  - 多种完成方式
  - 重复任务

  ### 3. 没有自动目标监听

  任务不会自动知道：

  - 已去过天台3次
  - 已获得某道具
  - 智力达到50
  - QTE成功
  - 与角色对话5次

  现在需要剧本在检查点手动判断并调用 quest stage/done。

  ### 4. 任务状态和实际条件可能不同步

  例如玩家已经有任务道具，但剧本没有执行到检查行，任务日志仍然显示未完成。

  ### 5. 重新 start 会覆盖阶段

  重复执行 quest start 会把任务重新设置到初始阶段。

  ### 6. 修改任务阶段顺序可能破坏旧存档

  旧存档的阶段2，在新版任务定义中可能已经代表不同内容。

  ### 7. 没有奖励事务

  任务完成后需要剧本另外发：

  - 金钱
  - 属性
  - 道具
  - CG
  - 成就

  如果漏写其中一项，任务状态已经完成但奖励可能没发。

  ### 8. 没有期限和失败条件

  不能直接描述：

  - 12月前完成
  - 3天内提交
  - 压力超过90则失败
  - 进入下一章自动过期

  ### 9. 没有追踪、置顶和地图提示

  任务日志目前偏展示，没有完整的玩家导航功能。

  ## 潜力与改进方向

  1. 将任务升级为状态图：

  - 未接取
  - 可接取
  - 进行中
  - 可提交
  - 已完成
  - 已失败
  - 已放弃
  - 已过期

  2. 给每个任务建立目标列表：

  - 访问地点
  - 收集道具
  - 属性达标
  - 事件成功
  - 对话完成
  - 击败敌人
  - 累计次数
  - 日期到达

  3. 目标自动订阅系统事件。

  地图、商店、属性、小游戏和日期变化时自动推进目标。

  4. 支持多个目标：

  - 全部完成
  - 任意一个完成
  - 完成其中两个
  - 隐藏可选目标

  5. 支持分支任务。

  不同选择进入不同后续阶段，最终可以得到不同结果。

  6. 支持任务期限。
  7. 支持自动奖励和原子结算。
  8. 支持重复任务：

  - 每日
  - 每周
  - 每月
  - 每周目
  - 无限重复但奖励递减

  9. 支持任务链和前置条件。
  10. 支持任务追踪：

  - HUD置顶
  - 地图标记
  - 当前目标进度
  - 距离截止时间

  11. 支持任务完成大型弹窗和演出队列。
  12. 支持任务历史。

  即使任务完成或失败，也能在日志中查看完整经过。

  13. 支持旧存档阶段迁移。
  14. 保留兼容层：

  即使内部升级成复杂对象，仍可以同步一个简单的 任务_<id> Flag，保持现有剧本兼容。

  # 八、六套系统共同存在的主要风险

  ## 1. 字符串ID风险

  Flag、标签、章节、事件ID、任务ID和结果名都依赖字符串。

  应该尽早建立：

  - 自动补全
  - 查找引用
  - 安全重命名
  - 重复检查
  - 不存在检查

  ## 2. 命令知识分散

  新增命令通常需要修改：

  - Parser关键字
  - Runner执行逻辑
  - 编辑器Schema
  - 校验器
  - 中文显示
  - 状态重建
  - 存档

  容易漏改。

  长期应建立统一命令注册表，让命令的：

  - 名称
  - 参数
  - 校验
  - 编辑器控件
  - 执行器
  - 状态影响

  尽量来自同一份定义。

  ## 3. VNScriptRunner 继续膨胀

  当前它同时负责：

  - 解析后执行
  - 输入
  - 存档
  - 自动/快进
  - 菜单
  - 事件
  - 任务
  - 属性
  - 日历
  - 章节
  - 调试重建

  短期方便，长期会成为修改风险中心。

  ## 4. 存档兼容

  当前新增普通Flag很容易，但以下变化风险很高：

  - Flag重命名
  - 标签重命名
  - 章节重排
  - 命令位置变化
  - 任务阶段变化
  - 新增调用栈
  - 新增随机状态

  需要尽早加入存档版本和迁移机制。

  ## 5. 缺少自动化测试

  建议至少覆盖：

  - 条件表达式
  - Label/Jump
  - 章节切换
  - Choice可达性
  - Event结果匹配
  - Quest状态变化
  - Stat钳制
  - 存档往返
  - 旧存档迁移

  ———

  # 九、最值得新增的跨系统能力

  ## 1. 统一触发器与事件调度器

  这是最适合你游戏的下一层。

  定义触发条件：

  - 日期到达
  - 属性跨越阈值
  - 任务完成
  - 获得道具
  - 去过地点
  - 小游戏成功
  - 好感变化
  - 月末、周末、日末

  每个触发器还可以配置：

  - 优先级
  - 是否一次性
  - 冷却时间
  - 互斥组
  - 目标章节和标签
  - 是否立即触发
  - 是否等当前对话结束
  - 是否占用本月剧情名额

  这样主循环不需要越来越长的 if 链。

  ## 2. 统一效果列表

  让 Choice、Event、Quest和行动都可以使用同一种效果：

  - 修改Flag
  - 修改属性
  - 发道具
  - 扣道具
  - 推进任务
  - 解锁CG
  - 播放SE
  - 触发演出
  - 跳转剧情

  效果应当能够批量验证、预览和原子执行。

  ## 3. 确定性随机系统

  统一管理：

  - 随机事件
  - 行动评级
  - 战斗结果
  - 掉落
  - 技能检定

  并将随机状态写进存档。

  ## 4. 剧情事件队列

  如果同一个月同时满足：

  - 圣诞事件
  - 任务完成
  - 好感20
  - 主线章节

  不要直接连续播放四段剧情。

  应该进入队列，根据优先级每个周期播放一个或允许玩家选择稍后查看。

  ## 5. 全局调试控制台

  显示：

  - 当前章节和标签
  - 最近跳转路径
  - 最近Choice
  - 当前活动Event
  - 所有Flag
  - 所有Stat
  - 所有Quest
  - 待触发事件队列
  - 随机种子
  - 存档版本

  并允许测试人员一键模拟：

  - 进入12月
  - 好感设为50
  - 完成任务
  - 发放道具
  - 跳到结局

  ## 6. 无画面的剧情模拟器

  不启动Unity画面，直接执行剧情逻辑，自动遍历不同选择，检测：

  - 不可达结局
  - 无限循环
  - 缺少标签
  - 无可用选项
  - 永远无法完成的任务
  - 永远不会触发的事件
  - 属性上限导致条件不可达

  # 十、建议实施优先级

  ## 第一优先级：安全和调试

  1. 修正Choice“全部隐藏就全部显示、全部付不起就免费解禁”的兜底。
  2. Flag变化事件增加名称、旧值、新值、来源。
  3. 制作Flag/Stat/Quest实时调试面板。
  4. 增加全项目字符串ID检查和引用查找。
  5. 加存档格式版本。
  6. 为条件、跳转、属性钳制、任务状态写自动化测试。

  ## 第二优先级：表达能力

  1. 条件支持 &&、||、括号和简单算术。
  2. 支持跨文件限定标签。
  3. 支持 call/return 和参数。
  4. Choice支持多个条件、多个成本、多个效果。
  5. Event模块声明参数和结果。
  6. 统一效果事务。

  ## 第三优先级：养成玩法

  1. 日期升级到年/月/周/日。
  2. 统一事件触发器和优先级队列。
  3. 确定性随机系统。
  4. 行动定义资产。
  5. 周计划和逐日执行。
  6. 分级结果与统一结算弹窗。
  7. 角色关系系统。
  8. 自动任务目标。

  ## 第四优先级：大型项目能力

  1. 类型化状态和作用域。
  2. 可保存长事件与3D小游戏场景。
  3. 剧情流程图。
  4. 分支自动遍历。
  5. 存档迁移工具。
  6. 资源异步加载。
  7. 周目继承与永久解锁管理。

  最终建议不是推翻当前系统。最稳妥的路线是保留现有 DSL 和“万物皆 Flag”的兼容性，在外面逐层增加类型、验证、事
  件调度和专门服务。

  当前系统很适合“快速做出很多玩法”；下一阶段要解决的是“做出很多玩法以后，怎样仍然知道它们为什么触发、由谁修
  改、能否存档、会不会互相冲突”。这将决定它最终是一个灵活的原型工具，还是能够支撑完整商业游戏的制作系统。
